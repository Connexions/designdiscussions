Archive API Requirements

All Content

The API MUST...

  • be asynchronous.  
  • have a simple URL structure that has a small set of parameters. A small set of parameters means there will be less things that change in the URL in the future.
  • take an EPUB file as a payload.  The EPUB can include...
    • An HTML document - required
    • resources
      • images
      • Java applets
      • small video files
      • audio files
      • labview files
      • other documents such as Word, PDF, etc.
  • return a payload acknowledgement message when payload successfully received. Payload acknowledgement should include the link for polling the status of the publish. The acknowledgement could include an HTTP status.
  • extract metadata about the module from the HTML document. Metadata will be stored as the OPF file in the EPUB.
  • perform validations.
    • payload has required contents
    • publisher is a valid CNX user
    • Links in source are valid
    • user listed in Roles are valid CNX users
  • return an error if validations fail.  
    • Validations that should fail immediately:
      • Incomplete payload
      • User IDs unknown
      • Security violations
    • Validations should be run and a complete list returned.  
      • Links in source are not valid
    • Errors should return an HTTP code, an API error code and a description of the problem.
  • On success, return a success message that includes some detail about what was created
    • URL of published content
    • Resources added
    • URL to poll status of post-publication jobs??

New Content

The API should …

  • perform all of the items listed for All Content
  • insert resources into storage.  IDs should be captured for later use
  • parse HTML for resources and update with new storage IDs
  • insert HTML into storage
  • mark HTML as latest version
  • return a message containing the URL to the newly published content and a list of resources added during the publish.
  • trigger export generation and post-publication jobs

Updated Content

Updated content payloads should only contain the files that have changed, but will accept (and ignore) existing resources.

The API should ...

  • perform all of the items listed for All Content
  • check resource IDs against previous version resource IDs.  This will discover which resources are being updated if any.
  • insert any new resources into storage.  The IDs should be saved for later use.
  • parse HTML for new resources and update with new storage IDs
  • insert HTML into storage
  • mark HTML as latest version
  • return a message containing the URL to the newly published content and a list of resources added during the publish.
  • trigger export generation and post-publication jobs

Derived Content

The API should …

  • perform all of the items listed for All Content
  • insert any new resources into storage.  The IDs should be saved for later use.
  • parse HTML for new resources and update with new storage IDs
  • insert HTML into storage
  • mark HTML as latest version
  • mark version as derived from other content return a message containing the URL to the newly published content and a list of resources added during the publish.
  • trigger export generation and post-publication jobs

Collections

Collections are handled in a similar way as listed in the requirements above with a couple of exceptions

  • Collections can contain published and unpublished content
  • There will be a need for several passes through all of the HTML as each of the resources and modules are added to storage.  This is to handle linkage across modules in the collection.

Polling API

The API should …

  • allow other systems to request the status content being published
  • limit number of times a requester can poll
  • blacklist the requestor if they violate the limitations
  • return a message that includes...
    • current action
    • previous actions as part of publishing the content

Logging

Some of the logging listed below could expire after a period of days since it is mainly being kept for troubleshooting.

The API should log …

  • request received
  • Create a UUID for each Request, that is then included in each processing step to make troubleshooting simpler
  • any messages returned to the requestor
  • each event as part of the publish
