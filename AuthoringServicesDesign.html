Authoring Requirements

NOTE: The mockups in this document are to display possible functionality, not how the site will actually look.  They are crude whiteboard or Libre Office drawings.  The actual site will look much different and much more polished.

Overview



Authoring has the following Components

Editor

The editor is a JavaScript-based editor (Aloha)

Use Cases

  • User should be able to edit content in a mostly WYSIWYG editor  This includes
    • figure editing
    • math editing
    • list editing
    • link editing
    • Exercise editing
    • Example Editing
    • Glossary Editing
    • The remaining CNXML elements

Use Cases

The user can

  • edit Connexions content in a mostly WYSIWYG way
  • Drag and drop block components
  • edit metadata

Requirements

  • Be as WYSIWYG as possible
  • Elements that cannot be totally WYSIWYG should be popups. 


  • Math Editing
    • Support will be provided for editing math as
      • MathML
      • LaTeX
      • ASCIIMath
    • Math imported via the LaTeX importer will remain in LaTeX for editing
      • For display, the math will be converted to MathML
    • Editor will open a popup to edit Math.

  • Preview
    • Will have PDF and Web Preview
  • Imports
    • User can import from the editor
      • Word
      • LaTeX
      • HTML
      • CNXML
      • Zip file
  • Publish
    • Should allow user to publish to Archive (Published content repository)
    • Publish should trigger generation of
      • PDF
      • EPUB
      • Complete Zip
      • Export HTML
      • Export CNXML
    • User should be able to see the status of the triggered events
  • Authentication
    • Users will be authenticated via OpenStax Accounts

Authoring

Use Cases

The user can

  • open a tab to edit metadata.  The metadata is saved when the user clicks on the Save button.
  • open a popup to publish a module.  When user selects to publish, they are returned to the Workspace page where they can see the publish updates..
  • open a tab to edit Roles. The roles are saved once the user clicks on the Save button.
  • trigger the generation of a preview PDF
  • switch to Webview preview

Client

  • retrieve module selected from workspace when client is opened
  • save a module to Authoring when user clicks the Save button

Mockups

This is a series of mockups that describe how the different editor features would work.

The Editor has several features

  • The Sve Draft button cnverts to a Publish button when content is saved
  • Metadata tab allows access to editing module metadata and Roles
  • Tools tab contains the Webview Preview option
  • Revert returns to Module to the previous edited version

The Import popup allows the user to

  • select the file to import
  • click the import button to import the file
  • System will determine the file type and handle correctly
  • If file type is not supported, the user will be notified and the type logged so we have a record of what people are trying to upload for future development.
Once the import is completed, the popup closes

The Publish popup will

  • require the user to enter a description
  • have similar license text from the current publish tab

System will perform some validations before publish is allowed

  • Title
  • CNX links exist

Once the content is sent to the published repository, the user will be sent to the Workspace view where they can see the status of the export generation.

If a role request or validation is preventing publishing, the user should be notified without opening the publish popup.

  • Files will be added as part of creating a link, image or figure. 
  • User will have option of uploading an image at that point.
  • Any files not part of the document will be removed when the module is published.
  • When images are imported, they must be part of a document that is part of the import.

The Roles section will

  • display current assigned roles
  • allow user to search for users by name or user id
  • close section when done.
  • display the status of each role request

When a user is using an OpenID, the workflow is

  • User A added User B to a module using their email address (OpenID)
  • User B is sent an email
  • User B must login to CNX using their OpenID
  • User B then confirms that they wish to be added to module

The Metadata section will

  • display a list of languages
  • display a list of previously used keywords
  • section will close when user is done.

Logging

The Editor should log events such as

  • Toolbar usage
  • Saving content
  • Usage of items from Tools dropdown
  • Button clicking on popups

Collection Editing

Data

  • Collection is a Hierarchy of links
  • Data needed for editing: module id, module title, link to module and subcollection titles
  • Collections stored as HTML
  • Modules stored as a nested list of links ( <ul><li><a href="m123" class="xinclude autogen-children">Title...)
  • See details in this Google Doc
  • URLs to modules will be relative
  • Collection titles will be stored the same as modules

Collection HTML Example

<div class="abstract"></div>
<ul>
  <li><span class="title">Subcollection</span>
    <ul>
      <li><a href="[UUID]@12" class="xinclude">Overridden Title</a></li>
      <li><a href="m9003" class="xinclude autogen-title">module</a></li>
      <li>
        <a href="m9003" class="xinclude autogen-title">module</a>
        <ol class="links">
          <li><a href="http://yahoo.com" data-strength="3" data-type="prerequisite">yahoo</a></li>
        </ol>
      </li>
    </ul>
  </li>
</ul>

Editing

  • Collection editing will be similar to viewing a collection.
  • The Table of Contents will allow drag and drop movement of subcollectins and modules.
  • The Table of Contents will also allow the creation of modules and subcollections
  • When collection is saved, the entire document will be sent to the server

Other

  • The following will be handled the same as with modules
    • metadata - Json
    • ACL
    • Publishing
    • Saved URLs

Versions

  • Collections need to have major and minor version numbers.
  • Major for when the collection structure is modified and minor for when collection content is modified.
  • Editions can be part of metadata

Workspace

  • Should have a Google Drive (Docs) type display


Workspace Requirements
  • A module is only listed if it is not in a Collection.
  • The same document can be in different folders for different users
  • The user should be able to search their Workspace for a document.
  • Modules or collections can be created via the Add button
  • Users can delete content from their Workspace. Deleting removes their access rights, but does not delete the document unless they are the last user with access.

Logging

Workspace should log events such as

  • Creating new items
  • Opening a document
  • Deleting an item

Authoring

Authoring is used to store content and media until it is published.  It has the following requirements.

  • Store some content on the file system and some in a database
    • Modules
      • Store Json in database
      • Metadata and content are in the same Json document
      • ACL information is stored with User information in a separate database
      • Creation date
      • Last modified date
      • User ids only are stored in the metadata. Other info in Authentication Services.
    • Collections
      • Store Json in database
      • The list of Colection contents will be an HTML document that is part of the Json
      • The list of contents will also be stored in a database table with collection id being the key
      • Metadata and content are in the same Json document
      • ACL information is stored with User information in a separate database
      • Creation date
      • Last modified date
      • User ids only are stored in the metadata.  Other info in the Authentication Services.
    • Folders
      • List of content
      • folders to filter content
      • Share content with others via applying roles
      • Store Json in database
      • List of content in a Folder will be stored as an HTML document in the Json
      • Folder contents will also be stored in a database table with folder id being the key
      • Metadata and content are in the same Json document
      • ACL information is stored with User information in a separate database
      • Creation date
      • Last modified date
  • Repository will have an API for performing CRUD operations
    • API should log access via API

Repository Use Cases

  • User can add a record
  • User can delete a record
  • User can update a record
  • User can retrieve a record
  • User can store media files
  • User can delete a media file
  • User can update metadata for a module or collection
  • User can update roles
  • User can create roles (happens when a module is created)

Comments

  • The Module Abstract will be moved to become part of the text displayed in the Module Editor.  This will allow the use of the Aloha editor for abstract editing. The Abstract will be stored as part of the Module HTML
  • All metadata that is not needed by the editor has been removed.  The missing metadata will be added when the content is published in the archive.
  • The fields institution, courseCode and instructor will most likely go away.
  • User access to a module will not be stored in the metadata, but as part of the Authentication Services.

Sword Interface

The Sword interface should work in the same manner as the current Connexions interface works.

  • A discovery service should be available that will return URLs based on user credentials
  • Create and update modules and collections will be allowed
  • Payload will be a zip file that can contain any of the file types Connexions can upload.
  • Error handling will continue the way it is currently implemented.


ACL Data


ACL data should be stored in a separate database from the Unpub Repo. This is needed because this information is shared by the Unpub Repo and the Archive. Data should be stored similar to current Connexions database schema. Some of the tables referenced below could be scaled back with less fields. See original Connexions Schema at https://github.com/Rhaptos/Products.RhaptosModuleStorage/blob/master/Products/RhaptosModuleStorage/sql/db_schema.sql

  • ACL data can be stored in the same database server as the Unpub Repo data
  • Unpub Repo and ACL can have separate schemas or they can be one schema with separate APIs for different data.

Business Logic

  • Code reviews should keep the logic in stored procedures in check
  • Stored Procedures will be written in a modular way so that they are easy to update
  • Code for Stored Procedures will be checked into Github
