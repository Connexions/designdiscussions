Transformation Services Requirements

General


Transformation Services should import

  • Word Documents
  • Zip files containing
    • HTML and resource files
    • CNXML and resource files
    • Word document
  • Latex zip files
  • CNXML files
  • HTML files

Transformation Services should create these export files
  • PDF
  • EPUB
  • Mobi
  • CNXML
  • HTML
  • Complete Zip

Each of the imports and exports should...
  • reuse as much of the existing Connexions code as possible
  • run as a separate service that can be updated, stopped and started without interfering with the other processes.
  • store generated files for a limited time such as 24 hours
  • perform validations before processing
    • required information is included
    • payload is valid

  • return an error if validations fail.  
    • Validations that should fail immediately:
      • Incomplete payload
      • required info is missing
      • Security violations
      • import/export process failure
    • Errors should return an HTTP code, an API error code and a description of the problem.


Messaging Queue


All requests should be received by a messaging queue.  Requests are picked up off of the queue and dispatched to the correct API for processing (import, export).  The queue should store messages so queue failure does not result in lost requests.  Access to the messaging queue is restricted by an identifying key.


Export API


The Export API should be a separate process from the rest of the services.  It should...

  • accept a request to generate any of the files
  • return a success message when the package is received.  It should include a URL that can be used for Polling the status of the job.
  • allow generation of all export files in one request.
  • allow generation of a PDF for previewing
  • deal with dependencies for file generation

A request should include
  • What to generate
  • Where the data is


Transform API


The Transform API should be a separate process from the rest of the services.  It should...

  • accept a request to import a file
  • determine what type of import based on file type sent by the content-type header.

  • return a success message when the payload is received.  It should include a URL that can be used for Polling the status of the job.
  • return another success message when the job is completed.

A request should include
  • the file to import


Monitoring


Each service should be monitored separately.  This can be done using Syslogs and Graphite or with some other option.


Logging


Some of the logging listed below could expire after a period of days since it is mainly being kept for troubleshooting.

The API should log …

  • request received
  • Create a UUID for each Request, that is then included in each processing step to make troubleshooting simpler
  • any messages returned to the requestor
  • each event as part of the publish
    • validation checks
    • generating files
    • storing files
    • importing files
    • returning files
  • initial polling request from a requestor
  • any lockout of a polling requestor


Polling API


The API should …

  • allow other systems to request the status files being generated or imported
  • limit number of times a requester can poll
  • blacklist the requestor if they violate the limitations
  • return a message that includes...
    • current action
    • previous actions as part of publishing the content
  • When job is completed, the polling URL should return the generated export file or a zip of the generated HTML and resources to import into the editor repository.
