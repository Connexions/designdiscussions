  • repository should be responsible for rewriting all of the URLs that reference other modules or other resources (images, etc.)

Phil thinks the unpublish repository API should be a subset of the Published content repository API. Ross and Ed disagreed. The point is to make the best API for each repository, not something generic from the start. if it is possible and logical for them to be the same, fine, but that should not be a design goal.

Kathi agrees with Phil about the API.

Ross is concerned that the Unpublish Repo API will be too heavyweight for its purposes. There was a discussion about how the unpublished repo should be saving parts of HTML instead of an intire document. Ed brought that discvussion to a halt because it has already been decided that we are storing the HTML on the file system which means we should be saving the entire document in version 1.0.

Zip vs Multipart

Ross suggested we use multipart messages rather than a zip file. Kathi brought up the fact that her team had trouble implementing multipart when testing the Sword interface from her OERPub tool. Zip files are easier to deal with on the command line.

Publishing a module that references another module

We talked through the problem of a user publishing a module that has links to an unpublished module. The Published repo will return an error if this happens. Having the editor do that check and allowing the user to select to publish both modules at once is not a version 1.0 operation.

Api should return some structure that has URLs to published content.

Paul suggested we use UUID which would solve some of the rewrite problem. We will discuss this further on Wed.

API - Publishing - Resource Handling - Push/Pull

  • API should be separate from under the hood behaviors of REPO
  • barebone REPO discussion
  • data contracts of repo
  • resource storage

Mapping

  • publish is potentially a multi-step, async process
  • repo does not edit or even parse content - bits served will be bits provided by editor
  • Editor POSTs content-to-be-published to same URL
  • includes session key
  • provides mapping of referenced contentids
  • repo never changes bits, we could include crypto signatures as validation.
  • mapping could include ‘pull these bits from this public URL’ as different flag
  • resource map is critical: separates repo from editor at the modular component level. It is essentially a dictionary, mapping
  • as an archive, the repo only provides the bits-as-deposited. If we want partially-already-processed content, then the webview could provide that as a service - the html w/ the map already applied, as it were.
  • the work involved is to parse a map (dictionary) first, and whenever hitting a resource-key in the body content, reflect it through the map to create the actual URL to fetch.

Mapping Alternative

  • feature of signing
  • if the POST in step 1 contains a set of relative paths (like '?url=m9003.html&url=m9004.html') then those are looked up in the signed ZIP payload
  • otherwise, they are remotely GET'd. Then, the 'signed' zip could be throught of as a webserver *content has to be parsed to be verified

Key Concepts & Concerns from Team Members:

  • mapping not needed in editor : only need resources mapped to module not to repository id names should be avoided
  • editor and REPO are separate components, if id's are shared then they are coupled and these components are not separate defeating our modular design approach
  • local names are no longer needed
  • mapping should be done for other Editors.
  • core feature for main documents with accessible resources
  • minimal should be for archive to store info and send back (file system like storage)
  • Currently our repo acts like a web server with relative urls to access resources.Archive isn't an HTML tree.
  • Build archive not tied to particular format.REPO only needs to map core docs and associate resources. Mapping be a document. Path to resources stored in archive. HTML documents have associated resources[pre-store resources hangs up Editor and creates a garbage issue] Id's will not conflict. Id's are only used as keys for map unique to document.

Two options: either REPO does little work to look within HTML. A mapping document would be needed to convert resourses with URLs. Other options: remove mapping document and change url's within HTML[editor side]

  • dont have to change original authoring bits with map.
  • map shouldn't be part of doc to guarantee bits are that of original author
  • WebView REPO/HTML. Cross referencing modules, collections, resources is an issue that needs resolving.
  • no link validation at Publish
  • don't fetch urls from external sources at publication.
  • validation solution: Editor layer should check two things: is this a valid url? local url for local image?

Resources Concepts & Concern Cont.

  • publish resources as modules and collections are published
  • map isn't convenient for other services/editors - too Connexions specific
  • fetch original bits from webview/repo. Browser already knows how to pull resources via URL. *either REPO or Editor needs to rewrite bits of resources. Why not have a short lived draft prior to rewrite. Rewrite must happen.
  • all resources do not need to live in REPO should be made publicly, easily accessible.
  • REPO has API then implements a services then REPO will have to change the bits.
  • Resources and copyrights are not about bits, it is about semantics.
  • Store separate metadata for resources and documents. Part of API should be able to send documents and resources. Editor wants low resolutions image. Repo stores highest resolution images. Concerned with pushing all the bits at a time. Example of collection .zip package for API. Go to Physics EPUB as example.

.zip Payload Case

  • author signs off on unpublished bits by pushing up a .zip file via Editor serverside
  • How does Editor rewrite resources from REPO when Author signs off content if bits are not to be rewritten?
  • Editor could go to REPO and grab module id - defined API hooks. Or REPO could take .zip file with relative links and publish and replace links.
  • Level 1: relative links.
  • Level2: walks HTML pulls links and anchor tags. REPO valiates then Editor would get url's pointing to REPO content. When editing Editor pulls HTML with public links to published content and an href of html points to REPO. When resources are changed, resources are not editable they are create a new.

Epub/PDF

  • Editor is only required to go to Published Repo and grab module id.
  • Module id is unique. Broken links should be validated via Editor.
  • Image resizing: during PDF/EPUB generation. Services to REPO should validate resource.

    -Must use strong algorithm. If one can attack database, can attack salt.

    -Standard - keep salt next to hash.

    -Storing salt next to hash prevents attack against all, but not against one individual.
